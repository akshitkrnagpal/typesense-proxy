# tsproxy

> **Early release вҖ” APIs are stabilizing but may still change between minor versions.**

A search proxy framework for [Typesense](https://typesense.org) with caching, rate limiting, an ingestion queue, and headless React components.

## Why tsproxy?

Typesense is a fast, typo-tolerant search engine. But exposing it directly to frontend clients creates challenges:

- **API key exposure** вҖ” Typesense search-only keys still reveal your cluster details
- **No caching layer** вҖ” every keystroke hits your search cluster
- **No rate limiting** вҖ” bad actors can overwhelm your search backend
- **No ingestion queue** вҖ” bulk writes can spike resource usage
- **InstantSearch compatibility** вҖ” Typesense needs a translation layer for InstantSearch widgets

tsproxy solves all of these with a single proxy server and CLI.

## Quick Start

```bash
npx @tsproxy/cli init
docker compose up -d
npx tsproxy dev
```

That's it. Three commands and you have a running search proxy.

## Packages

| Package | Description |
|---------|-------------|
| [`@tsproxy/cli`](/packages/cli) | CLI вҖ” `init`, `dev`, `start`, `migrate`, `seed`, `health` |
| [`@tsproxy/api`](/packages/api) | HonoJS proxy server with search, ingest, caching, rate limiting |
| [`@tsproxy/js`](/packages/js) | InstantSearch-compatible `searchClient` adapter |
| [`@tsproxy/react`](/packages/react) | Headless React components with BaseUI-style overrides |

## CLI Commands

```bash
tsproxy init         # Interactive project setup
tsproxy dev          # Start proxy in dev mode (hot reload)
tsproxy start        # Start proxy in production mode
tsproxy build        # Build for production
tsproxy seed <file>  # Seed data via the ingest API
tsproxy migrate      # Sync Typesense schema with config
tsproxy health       # Check Typesense + Redis status
```
